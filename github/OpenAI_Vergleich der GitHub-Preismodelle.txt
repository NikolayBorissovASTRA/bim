Vergleich der GitHub-Preismodelle und Empfehlung für ASTRA

Übersicht der GitHub-Preismodelle und Funktionen

GitHub bietet mehrere Pläne mit unterschiedlichem Funktionsumfang und Kosten. Für Organisationen sind insbesondere drei Hauptoptionen relevant: GitHub Free, GitHub Team und GitHub Enterprise (Enterprise gibt es als Cloud-Service oder als selbstgehostete Server-Version). Im Folgenden werden diese Modelle mit ihren wichtigsten Features, Einschränkungen und Kosten gegenübergestellt, um eine fundierte Entscheidung für das Bundesamt ASTRA zu ermöglichen. Dabei werden Kosten-Nutzen-Aspekte hervorgehoben ￼ ￼.

GitHub Free (kostenloser Plan)
	•	Kosten: $0 (kostenfrei).
	•	Einsatzbereich: Geeignet für kleinere Teams oder Einzelentwickler, die grundlegende Funktionalität benötigen und mit Open-Source oder kleinen internen Projekten arbeiten ￼.
	•	Funktionen: Ermöglicht unbegrenzt viele öffentliche Repositories mit vollem Funktionsumfang sowie unbegrenzt viele private Repositories mit eingeschränktem Funktionsumfang ￼. Inkludiert sind Community-Support, einfache Teamverwaltung und 2.000 GitHub Actions CI/CD-Minuten pro Monat für Automatisierungen ￼.
	•	Limitierungen: Erweiterte Kollaborationsfunktionen sind in privaten Repositories eingeschränkt. Insbesondere fortgeschrittene Branch-Schutzregeln (z.B. verpflichtende Code-Reviews durch mehrere Personen, Statusprüfungen vor Merge) stehen im Free-Plan nicht zur Verfügung ￼. Auch Code Owners (automatische Zuweisung von Reviewern für bestimmte Dateipfade) und einige Sicherheits-Features fehlen im Free-Tarif. Speicher für Packages ist auf 500 MB begrenzt ￼. Für ein professionelles Team wie ASTRA ist der Free-Plan meist nicht ausreichend, da wichtige Governance-Funktionen fehlen.

GitHub Team (Bezahlplan für Teams)
	•	Kosten: ca. $4 pro Benutzer/Monat (pro seat) ￼ – für z.B. 30 Nutzer $120/Monat ($1.440 pro Jahr), für 70 Nutzer $280/Monat ($3.360 pro Jahr). (Abrechnung erfolgt jährlich pro User.)
	•	Einsatzbereich: Empfohlen für wachsende Teams oder Organisationen, die erweiterte Kollaborations- und Verwaltungstools benötigen, aber noch keine umfassenden Enterprise-Integrationen voraussetzen ￼.
	•	Funktionen: Enthält alles aus dem Free-Plan plus wichtige Zusatzfunktionen für private Repositories: z.B. geschützte Branches, verpflichtende Pull-Request Reviews (ein oder mehrere Reviewer müssen Änderungen freigeben), Team- und mehrere Reviewer pro Pull Request, Code Owner Dateien, u.a. ￼. Dadurch können Code-Änderungen in ASTRA immer von Kollegen geprüft werden, bevor sie in den Hauptzweig gelangen. Außerdem inbegriffen: E-Mail Support von GitHub, 3.000 GitHub Actions CI/CD-Minuten pro Monat für das gesamte Team (entspricht 50 Stunden Build/Workflow-Laufzeit) sowie 2 GB Speicher für GitHub Packages (z.B. Container oder Bibliotheken) ￼ ￼. Optional kann auf Team-Ebene schon GitHub Advanced Security hinzugebucht werden (für Code-Scanning und Secret-Scanning), falls notwendig ￼. Wikis und Projekt-Insights (Traffic, Commits, etc.) stehen ebenfalls in privaten Repos zur Verfügung.
	•	Limitierungen: Keine SSO/Verzeichnis-Integration: Die Verwaltung der Benutzer erfolgt manuell über GitHub (kein SAML Single Sign-On mit dem unternehmensinternen Identity-Provider im Team-Plan) ￼. Audit-Logs und Compliance-Features sind rudimentärer als im Enterprise-Plan. Private GitHub Pages (z.B. interne Websites/Docs hosten) sind im Team-Plan nicht möglich – GitHub Pages kann hier nur für öffentliche Projekte genutzt werden ￼. Insgesamt fallen pro Benutzer Gebühren an, d.h. die Kosten steigen linear mit der Teamgröße ￼. Für ASTRA mit 10–70 Nutzern bleiben die Kosten jedoch moderat im Vergleich zum Enterprise-Plan.

GitHub Enterprise Cloud (Enterprise Cloud-Plan)
	•	Kosten: ca. $21 pro Benutzer/Monat (Listenpreis) ￼. Oftmals wird eine Mindestanzahl von ~50 Nutzern für Enterprise-Verträge vorausgesetzt ￼. Für 50 Nutzer entspricht das rund $1.050/Monat (~$12.600/Jahr); bei 70 Nutzern $1.470/Monat ($17.640/Jahr). Enterprise-Kunden können u.U. Rabatt oder Jahresverträge verhandeln. Hinweis: Für verifizierte Non-Profit-Organisationen gewährt GitHub alternativ kostenlose Team-Pläne bzw. 25% Rabatt auf Enterprise ￼ ￼, was für ASTRA (Bundesbehörde, keine Non-Profit im engen Sinne) vermutlich nicht zutrifft.
	•	Einsatzbereich: Konzipiert für größere Organisationen oder solche mit erhöhten Sicherheits- und Compliance-Anforderungen ￼. Bietet umfangreiche Verwaltungs- und Integrationsmöglichkeiten, sinnvoll wenn z.B. Einbindung in ein zentrales Identity-Management (AD/LDAP/SSO), erweiterte Auditierung oder strengere Richtlinien nötig sind.
	•	Funktionen: Enthält alle Features des Team-Plans und erweitert diesen um Enterprise-Funktionen ￼:
	•	SAML Single Sign-On (SSO) und SCIM: Integration mit einem externen Identitätsprovider (z.B. SwissID, Azure AD, etc.) zur zentralen Verwaltung der Benutzer und automatischen Berechtigungssteuerung ￼. Dies erhöht die Sicherheit und vereinfacht das On/Off-Boarding: Entfernt man einen Nutzer in der zentralen Benutzerverwaltung, wird sein Zugriff auf GitHub automatisch entzogen (SCIM) ￼ ￼.
	•	Erweiterte Sicherheits- und Compliance-Kontrollen: z.B. erweiterte Audit-Logs (Aktivitäten der Nutzer über API abrufbar und länger aufbewahrt) ￼, Audit Log Streaming in externe Systeme, IP-Zugriffsbeschränkungen (Allow Lists) ￼, Repository Rules unternehmensweit erzwingbar (zentrale Richtlinien für Branchschutz, Dateitypen, Commit-Signaturen etc.) ￼. Außerdem Umgebungs-Schutzregeln für GitHub Actions Workflows (z.B. Freigabeprozess bevor Deployments in bestimmte Umgebungen laufen) ￼.
	•	Großzügige Inklusiv-Ressourcen: 50.000 CI/CD-Minuten pro Monat (833 Stunden) für GitHub Actions in privaten Repos ￼ –  deutlich mehr Puffer für Builds/Automation als die 3.000 im Team-Plan. Zudem 50 GB Packages-Speicher inklusive ￼ (für Container-Images, Artefakte etc.).
	•	Verfügbarkeit und Support: Vertraglich zugesicherte 99,9% Uptime (SLA) der GitHub-Plattform ￼ sowie priorisierter Enterprise Support (schnellere Reaktionszeiten, dedizierte Ansprechpartner bei Bedarf). Für noch höhere Ansprüche kann Premium Support hinzugebucht werden (24/7 Hotline, 30-Minuten-Reaktionszeit bei dringenden Tickets) ￼ ￼ (gegen Aufpreis).
	•	Enterprise-Account und Zentralverwaltung: Möglichkeit, einen übergeordneten Enterprise-Account zu führen, unter dem mehrere Organisations-Accounts von ASTRA verwaltet werden könnten ￼. Dies vereinfacht Verwaltung, falls z.B. mehrere Unterorganisationen oder Projekte jeweils eigene GitHub-Orgs hätten (Lizenzkosten werden für gleiche User nur einmal gezählt). Für ASTRA als einzelne Organisation ist das weniger relevant, aber der Enterprise-Plan erlaubt optional die Anlage interner Repositories: Repos, die innerhalb aller Mitarbeiter von ASTRA (bzw. der Enterprise-Org) einsehbar sind, jedoch für externe nicht sichtbar ￼. Dies kann nützlich sein, wenn ASTRA mehrere Organisationen auf GitHub hätte oder mit anderen Bundesämtern im selben Enterprise-Verbund Code teilt.
	•	Datenresidenz & Compliance: Seit Ende 2024 bietet GitHub Enterprise Cloud die Möglichkeit, Daten in der EU-Region zu hosten ￼. Für ASTRA bedeutet das, Code und Repository-Daten könnten auf Wunsch innerhalb EU-Rechenzentren gespeichert werden, was Datenschutzbedenken minimiert. Zudem ist GitHub Enterprise Cloud nach gängigen Standards zertifiziert (u.a. SOC1/SOC2 Type 2 Berichte ￼, FedRAMP Tailored ATO für US-Behörden ￼). Für Schweizer Bundesbehörden kann die EU-Datenhaltung und diese Zertifizierungen ein wichtiger Aspekt hinsichtlich Compliance sein.
	•	Optional: Advanced Security & weitere Add-ons: Auf Enterprise-Ebene lassen sich erweiterte Sicherheitsfunktionen hinzubuchen, z.B. GitHub Advanced Security (GHAS) für CodeScanning (Erkennung von Sicherheitslücken im Code) und Secret Scanning (Erkennung von Geheimnissen wie Passwörtern in Commits). Dieser Zusatz kostet allerdings etwa $19 pro User/Monat extra ￼ ￼ – fast eine Verdoppelung der Kosten – und lohnt sich nur, wenn ASTRA strenge Code-Security-Compliance erfüllen muss. Ebenso verfügbar (gegen separate Gebühren) sind GitHub Copilot (KI-Codierassistent) oder Codespaces (Cloud-Entwicklungsumgebungen), was aber für ASTRA derzeit optional ist.
	•	Limitierungen: Höherer Preis pro Nutzer – das Preis-Leistungs-Verhältnis muss stimmen, sprich die zusätzlichen Funktionen sollten vom ASTRA-Team auch benötigt und genutzt werden. Für ein Team von <100 Personen ist der Enterprise-Plan eine erhebliche Investition, daher sollte ein tatsächlicher Bedarf an SSO/Compliance gegenüber dem Team-Plan abgewogen werden. Außerdem erfordert die Einführung von SSO/SCIM etwas Setup-Aufwand in Zusammenarbeit mit der IT (Identity-Provider Integration).

GitHub Enterprise Cloud eignet sich vor allem dann, wenn ASTRA zentrale Benutzerverwaltung (SSO), erweiterte Sicherheitsanforderungen oder sehr umfangreiche Automatisierungen benötigt. Für Teams unter ca. 500 Nutzern wird in der Regel die Cloud-Variante empfohlen, da diese sofort einsatzbereit ist und keinen Infrastrukturaufwand verursacht ￼.

GitHub Enterprise Server (selbstgehostet, On-Premises)
	•	Kosten: Enterprise Server Lizenzen basieren ebenfalls auf Benutzerzahlen (ähnlich $21/Nutzer/Monat Liste). Allerdings fallen zusätzlich Infrastrukturkosten an, da ASTRA die Server selbst hosten und betreiben müsste. Oftmals gibt es eine Mindestlizenz (z.B. 50 oder 100 Nutzer) und Verträge werden jährlich abgeschlossen. Beispiel: 50 User ~$12.600/Jahr Lizenzkosten, plus Hardware/VMs und Wartungsaufwand.
	•	Einsatzbereich: Gedacht für Organisationen, die strikte Datenhoheit benötigen oder Cloud-Dienste nicht nutzen dürfen (z.B. wegen rechtlicher Vorschriften) oder für sehr große Unternehmen (>1000 Entwickler) ￼. In solchen Fällen wird der GitHub Enterprise Server in der eigenen Umgebung (Rechenzentrum oder Cloud der Wahl) betrieben.
	•	Funktionen: Funktional identisch mit Enterprise Cloud hinsichtlich Benutzer-Features (SSO, Compliance, etc.) – d.h. ASTRA würde alle Enterprise-Vorteile erhalten, aber gehostet im eigenen Netz. Einige Cloud-bezogene Dienste (z.B. GitHub Actions) müssen entweder via GitHub Connect angebunden oder separat betrieben werden. GitHub Actions kann on-prem mit selbstgehosteten Runnern genutzt werden, allerdings gibt es in Enterprise Server standardmäßig keine GitHub-gehosteten Runner – man muss also eigene Runner-VMs bereitstellen.
	•	Aufwand/Nutzen: Deutlich höherer Betriebsaufwand. ASTRA müsste Server oder VM bereithalten, die Installation und Updates von GitHub selbst durchführen, Backups und Ausfallsicherheit sicherstellen etc. ￼ ￼. Dies erfordert IT-Personal und Know-how. Für ein Team von ~70 Personen dürfte der Total Cost of Ownership meist höher liegen als bei der Cloud-Lösung. Nur wenn Richtlinien eine externe Cloud absolut ausschließen oder besondere Integrationen auf Code-Ebene nötig sind, würde sich dieser Weg lohnen. GitHub selbst empfiehlt Enterprise Server eher für sehr große Organisationen oder spezielle Compliance-Vorgaben, während für kleinere Teams die Cloud-Variante wegen der geringeren operativen Komplexität bevorzugt werden sollte ￼.

Zusammenfassung der Optionen: Für ASTRA kommen GitHub Team oder Enterprise Cloud als realistische Optionen in Frage. GitHub Free bietet nicht die benötigten Funktionen (gerade bzgl. geschützter Branches und Reviews) und würde die Qualitätssicherung gefährden. GitHub Enterprise Server erscheint für die Teamgröße überdimensioniert, es sei denn, gesetzliche Vorgaben erzwingen eine Selbsthosting-Lösung. Im nächsten Abschnitt betrachten wir konkret die Anforderungen (Annahmen) von ASTRA und leiten daraus die optimale Wahl ab.

Annahmen und Anforderungen von ASTRA

Um das passende Preismodell zu bestimmen, betrachten wir ASTRAs voraussichtliche Nutzung von GitHub (basierend auf den gegebenen Annahmen):
	•	Teamgröße und Nutzer: Anfangs ca. 10–30 aktive Benutzer (Mitarbeitende), potenziell wachsend auf bis zu 70 Nutzer innerhalb eines Jahres. Die Lösung muss also skalieren können, ohne dass technische oder finanzielle Engpässe entstehen. GitHub Team und Enterprise lassen sich jederzeit um zusätzliche User erweitern (Team linear skalierend in Kosten, Enterprise ggf. stufenweise je nach Vertrag).
	•	Repositorien und Inhalte: ASTRA wird mehrere Repositories verwalten, um verschiedene digitale Artefakte abzulegen. Digitale Artefakte sind hier alle erstellten digitalen Inhalte wie Quellcode, Dokumentationen (z.B. Markdown-Dateien, PDF-Dokumente), Vorlagen (Templates, z.B. Formularvorlagen als JSON/XML), Diagramme (z.B. Mermaid-Diagramme in Markdown), Präsentationen (z.B. PowerPoint-Dateien) und Prozessbeschreibungen. Diese Artefakte unterteilen sich in:
	•	Interne Artefakte: Inhalte, die nur intern genutzt werden (nicht öffentlich). Beispielsweise interne Dokumentationen, behördeninterne Scripts/Code, Vorlage-Dokumente oder Präsentationen für den Behördengebrauch. Diese sollen in privaten Repositories liegen, zugänglich nur für ASTRA-Mitarbeiter.
	•	Öffentliche Artefakte: Inhalte, die ASTRA der Öffentlichkeit oder externen Partnern zugänglich machen möchte. Beispielsweise Open-Source-Projekte (Code von ASTRA entwickelten Tools), offene Daten oder öffentlich verfügbare Dokumentationen/Standards. Diese würden in öffentlichen Repositories verwaltet, sodass sie von Bürgern oder externen Stellen eingesehen und ggf. per Pull Request mitverbessert werden können.
	•	Branching und Workflow: Änderungen an den digitalen Artefakten – egal ob Code oder Dokument – sollen einem Review-Prozess unterzogen werden. D.h. Änderungen werden in einem separaten Feature-Branch vorgenommen und als Pull Request (PR) zur Aufnahme in den Hauptzweig (z.B. main-Branch) vorgeschlagen. Ein oder mehrere Reviewer prüfen die Änderung und geben sie frei, bevor sie in den Hauptzweig gemergt (integriert) wird. Dies gewährleistet Qualitätssicherung und Nachvollziehbarkeit (wer hat was geändert). GitHub ermöglicht dies out-of-the-box mit seinem PR-Workflow und schützbaren Branches. (Hinweis: Im Team-Plan kann man Reviews verpflichtend machen, im Free-Plan nicht) ￼.
	•	Continuous Integration (CI) & Automation: ASTRA plant, GitHub Actions für automatisierte Abläufe zu nutzen, aber in geringem Umfang („ja, aber wenig intensiv“). Das könnte etwa beinhalten: automatisches Ausführen von Tests für Code-Projekte bei jedem PR, ggf. das Bauen von Markdown-Dokumentation in PDF, oder Prüfen von Markdown/JSON-Syntax, etc. Schwergewichtiges CI/CD (wie große Build-Pipelines oder ständige Deployments) ist vorerst nicht vorgesehen. Somit sind die in den Plänen enthaltenen freien Minuten (Team: 3.000 Minuten/Monat, Enterprise: 50.000 Minuten/Monat) wahrscheinlich ausreichend. 3.000 Minuten entsprechen 50 Stunden Runner-Zeit – z.B. 100 Builds à 30 Minuten jeden Monat – was bei „wenig intensiver“ Nutzung nicht überschritten wird. Sollte das Kontingent doch einmal nicht reichen, kann man in GitHub Team zusätzliche Minuten zukaufen oder einzelne Runner selbst hosten ￼.
	•	Große Dateien: Artefakte wie Videos oder sehr große Binärdateien sollen nicht in Git gespeichert werden. Die durchschnittlichen Dateigrößen liegen bei ~3 MB (Markdown, PDF) bis 5 MB (PPT) pro Datei. Gelegentlich auftretende größere Dateien (über ~50–100 MB) würden extern gelagert. Dadurch vermeidet ASTRA, dass Repository-Klone zu groß werden oder Git LFS (Large File Storage) nötig wird. GitHub erlaubt zwar Dateien bis 100 MB im Repo, aber Best Practice ist, große Dateien (> ~50 MB) nach Möglichkeit fernzuhalten. ASTRA hält sich an diese Praxis, sodass der Speicherbedarf pro Repository moderat bleibt.
	•	Zugriffskontrolle & Sicherheit: Es wird Branchenschutz eingerichtet, insbesondere auf dem main-Branch der wichtigsten Repositories. Das bedeutet: direkte Commits/Pushes auf main werden untersagt, Änderungen müssen über PR kommen und mindestens eine Review-Genehmigung erhalten, bevor sie gemergt werden. Außerdem kann konfiguriert werden, dass z.B. main nur von bestimmten Rollen (Maintainer) gemergt werden darf. Dieses Regelwerk schützt die Integrität wichtiger Artefakte. Zudem soll für alle Benutzer die 2-Faktor-Authentifizierung (2FA) auf GitHub verpflichtend sein – GitHub Organisationen können das erzwingen, was unbefugten Zugriff auf Konten erschwert. In einem Enterprise-SSO-Szenario ist 2FA meist durch die IdP-Anmeldung abgedeckt, aber auch im Team-Plan kann man zumindest 2FA für die GitHub-Konten verlangen.

Diese Anforderungen im Hinterkopf, betrachten wir, welcher GitHub-Plan diese am besten abbildet.

Repository-Plan für interne & öffentliche Artefakte und Workflow

ASTRA sollte eine klare Repository-Strategie festlegen, um interne und öffentliche Inhalte sauber zu trennen und effizient zu verwalten. Konkret wird vorgeschlagen:
	•	Trennung von intern und öffentlich: Standardmäßig werden alle neuen Repositories zunächst als private erstellt (zugänglich nur für ASTRA-Mitglieder). Artefakte, die veröffentlicht werden sollen, können entweder in ausgewählten Repositories von privat auf öffentlich gestellt werden, oder von Anfang an in Public-Repos gepflegt werden. Wichtig ist eine Richtlinie: Nur nach Freigabe durch zuständige Stellen (z.B. Projektleiter) darf ein Repository öffentlich geschaltet werden. GitHub Team und Enterprise erlauben diese Umschaltung; in Enterprise könnte man auch globale Richtlinien setzen, die z.B. das Erstellen öffentlicher Repos nur Organisationseigentümern erlauben (zusätzliche Kontrolle).
	•	Struktur und Beispiele: Für interne Artefakte bietet es sich an, thematische Repositories zu erstellen. Beispiele:
	•	Repository “astra-intern-dokus”: Private Repo mit Markdown-Dokumentationen, Prozessbeschreibungen und Mermaid-Diagrammen zu internen Abläufen. Mit branchbasiertem Review, sodass Änderungen an Richtliniendokumenten immer gegengezeichnet werden.
	•	Repository “astra-vorlagen-intern”: Private Sammlung von Vorlagen (Word, JSON, XML Schemas, etc.) für interne Verwendung.
	•	Repository “astra-code-ProjektX”: Privates Repo für den Quellcode eines internen Tools “Projekt X”. Entwickler arbeiten hier kollaborativ am Code, Reviews stellen Codequalität sicher. Bei Bedarf könnte dieses Projekt später open-source gemacht werden.
	•	Repository “astra-public-website”: Öffentliches Repo, z.B. für eine Webseite oder Dokumentation, die ASTRA für Bürger bereitstellt (Inhalt könnte z.B. via GitHub Pages öffentlich gehostet werden).
	•	Repository “astra-open-source-toolY”: Öffentliches Repo für ein von ASTRA entwickeltes Tool Y, das quelloffen ist. Externe können den Code einsehen und evtl. via Pull Requests beitragen.
	•	Branching-Modell: Jedes Repository sollte einen Hauptbranch (i.d.R. main) haben, der den freigegebenen Stand repräsentiert. Für Änderungen wird ein Feature-Branch vom Hauptbranch erstellt – typischerweise benannt nach dem Feature oder Issue (z.B. feat/neue-doku-sektion). In diesem Branch nimmt ein Mitarbeiter die Änderungen vor. Anschließend eröffnet er einen Pull Request zurück in main. Der PR beschreibt die Änderung, ggf. mit Referenz zu einer Issue (Aufgaben-Tracking) und unterlegt mit automatisch generierten Diffs aller Änderungen.
	•	Pull-Request Reviews: Kollegen (Reviewer) werden dem PR zugewiesen (manuell oder via Code Owners automatisch, falls eingerichtet). Mindestens ein Reviewer muss den PR durchsehen und approve (genehmigen), bevor ein Merge möglich ist. Für wichtige Artefakte kann ASTRA auch zwei Genehmigungen verlangen (Team-Plan erlaubt mehrere Required Reviewers einzustellen ￼). Während des Reviews können Änderungswünsche (Änderungskommentare) angebracht werden, die der Autor im Branch umsetzt. GitHub erleichtert dies durch Inline-Kommentare im Diff und Konversation im PR.
	•	Merge und Deployment: Nach erfolgter Review und etwaigen Korrekturen wird der Pull Request in den Hauptbranch gemergt. Dieser Merge kann standardmäßig ein Squash-Merge sein (alle Änderungen des PR als ein Commit) oder ein normaler Merge-Commit, je nach vereinbarter Vorgehensweise. Der Hauptbranch ist damit aktualisiert. Optional kann anschließend ein Deployment oder eine Publikation erfolgen, z.B.
	•	Bei Code-Repositories: automatischer Build/Deploy über GitHub Actions (sofern eingerichtet).
	•	Bei Doku-Repos: evtl. Veröffentlichung via Pages oder Export der Markdown in PDF.
	•	Bei Vorlagen: neue Version wird organisatorisch verteilt.

Abbildung: Vereinfachter Workflow für Änderungen via Branch und Pull Request. Ein Entwickler erstellt einen Feature-Branch für seine Änderung, eröffnet daraus einen Pull Request, der von Kollegen im Code-Review geprüft wird. Nach deren Freigabe (Approval) wird die Änderung in den main-Branch gemergt und damit offiziell übernommen.

Diese Vorgehensweise gewährleistet, dass alle digitalen Artefakte – seien es Code oder Dokumente – einem Vier-Augen-Prinzip unterliegen und Änderungen nachvollziehbar dokumentiert sind. GitHub protokolliert sämtliche PRs, Kommentare und Commits, sodass für das Management Transparenz besteht, wer wann welche Änderung genehmigt hat. Insbesondere für Dokumentationen und Prozesse kann dies wichtig sein (Versionierung von Richtlinien etc.).

Weitere bewährte Best Practices für die Repository-Nutzung bei ~10–70 Nutzern:
	•	Teams und Berechtigungen: In der GitHub-Organisation sollten für ASTRA Teams definiert werden (z.B. Admins, Entwickler, Reviewer, Projekt X Team etc.), um Rechte zu verwalten. So kann man z.B. bestimmten Teams Schreibrechte auf Repository A geben, während andere nur Leserechte haben. Organisation-Owner (Admins) sollten nur wenige vertrauenswürdige Personen sein (z.B. Toolverantwortliche), da sie alle Rechte haben. Im Enterprise-Plan könnte man via SSO Gruppen aus dem bestehenden Verzeichnisdienst (z.B. Active Directory) direkt abbilden. Im Team-Plan erfolgt die Zuweisung manuell in GitHub.
	•	Branch-Schutzregeln einrichten: Für die Hauptbranches (main bzw. master oder z.B. develop sofern genutzt) sollten Branch Protection Rules aktiviert sein. Konkret:
	•	“Require pull request review before merging” – mindestens 1 (oder 2) Review(er) erforderlich ￼.
	•	“Dismiss stale reviews” – falls nach einer Freigabe noch Änderungen gemacht werden, muss neu genehmigt werden.
	•	“Require status checks to pass before merging” – falls CI-Tests laufen, müssen alle auf Grün sein.
	•	“Include administrators” – auch Admins müssen den Prozess durchlaufen (optional, aber gut für Vorbildfunktion).
	•	Ggf. “Require signed commits” – kann aktiviert werden, wenn ASTRA Wert auf kryptographisch signierte Commits legt.
	•	Automatisierung vorsichtig nutzen: GitHub Actions Workflows sollten eingesetzt werden, wo sie Mehrwert bringen – z.B. automatische Tests bei Codeänderungen, Linters für Formatierung, oder das Generieren von Dokumentationsartefakten. Da ASTRA die Actions-Nutzung als „wenig intensiv“ einschätzt, wird das Minutenkontingent im Team-Plan vermutlich genügen. Dennoch sollte man Monitoring der CI-Minuten betreiben. Bei drohendem Engpass können entweder zusätzliche Minuten zugekauft werden ￼ oder man definiert bestimmte Jobs auf self-hosted Runnern (z.B. ein eigener Server für Builds, falls vorhanden), die keine Minuten verbrauchen. In der Enterprise-Version ist der inkludierte Puffer ohnehin deutlich höher.
	•	Schulung und Konventionen: Alle Benutzer sollten im Umgang mit GitHub geschult werden, insbesondere diejenigen, die neu mit Git/GitHub arbeiten. Es sollten Konventionen definiert sein (und idealerweise schriftlich im Repo dokumentiert): z.B. Namensschema für Branches, wie Commits und PRs zu kommentieren sind, wie Reviewer auszuwählen sind, und innerhalb welcher Fristen Reviews erfolgen sollten, damit Projekte nicht stocken. Klare Review-Richtlinien (Was prüfen? Wie freigeben? Wann ablehnen?) helfen der Qualität.
	•	Issues und Projektmanagement: Es empfiehlt sich, für Arbeiten an Artefakten GitHub Issues zu nutzen, um Aufgaben, Fehler oder Change Requests zu tracken. Diese Issues können mit Pull Requests verknüpft werden (Schlüsselwort “Closes #X” in PR-Beschreibung), sodass man später sieht, welche Änderung welche Issue gelöst hat. Für das Management könnte man GitHub Projects (Kanvas-Boards) nutzen, um den Überblick über laufende Tasks zu behalten. Dies ist im Team-Plan bereits verfügbar und hilft bei der Priorisierung und Transparenz.
	•	Dokumentation und Wiki: Für interne Wissenssammlungen kann das GitHub-Wiki eines Repositories genutzt werden (im Team-Plan auch in privaten Repos verfügbar). Allerdings sind Wiki-Änderungen nicht per PR reviewbar. Alternativ kann man Dokumentation im Repository selbst versionieren (z.B. als Markdown im docs/ Ordner), was via PR überprüft würde. Da GitHub Markdown mit Mermaid-Diagrammen direkt rendern kann (Diagramm wird aus dem Text erzeugt und im Browser angezeigt ￼ ￼), können Architekturdiagramme oder Ablaufdiagramme direkt im Repo als .md gepflegt werden. So bleiben Diagramme stets in Sync mit der Doku und überprüfbar.
	•	Open Source Contribution Handling: Falls ASTRA öffentliche Repos hat, sollte definiert werden, wie externe Beiträge angenommen werden (Contributor License Agreement nötig? Code of Conduct für Community? etc.). GitHub bietet hierfür Vorlagen an (z.B. eine CONTRIBUTING.md Datei mit Beitragsrichtlinien). Für öffentliche Projekte kann auch das GitHub Discussions-Feature sinnvoll sein, um Feedback von externen Nutzern zu strukturieren.

Zusammenfassend stellt dieser Repository-Plan sicher, dass sowohl interne wie öffentliche digitale Artefakte effizient und sicher mit GitHub verwaltet werden. Die Nutzung von Branches und Pull Requests für alle Änderungen gewährleistet Qualität und Nachvollziehbarkeit. Durch die klar getrennten Repositories nach Verwendungszweck behält ASTRA die Kontrolle darüber, welche Informationen intern bleiben und was nach außen gegeben wird.

Empfehlung: Geeignetes GitHub-Preismodell für ASTRA

Nach Abwägung der Features, Kosten und der spezifischen Anforderungen von ASTRA empfehlen wir folgendes Vorgehen:

Start mit GitHub Team-Plan, mit Option auf Enterprise Cloud bei Bedarf.

Der GitHub Team Plan erfüllt bereits die meisten der identifizierten Anforderungen mit geringeren Kosten: ASTRA erhält unbegrenzt private Repos für interne Projekte und kann öffentliche Repos für Open Source anlegen. Branchschutz, Code-Review-Pflicht und Code Owners – zentrale Funktionen für Qualitätssicherung – sind im Team-Plan enthalten ￼. Die Teamgröße von zunächst 10–30 (später bis 70) Nutzern ist problemlos im Team-Tarif handhabbar und kostenseitig überschaubar (im Maximum um die 3.360 USD/Jahr für 70 User). Die CI-Kapazität von 3.000 Minuten dürfte für die anvisierte „wenig intensive“ Nutzung ausreichen. Sollte mehr benötigt werden, kann ASTRA entweder Minuten zubuchen oder im Notfall auf Enterprise upgraden – die Investition ist also vorerst gering, und man zahlt nur mit zunehmendem Nutzen mehr.

Wann wäre GitHub Enterprise (Cloud) anzuraten? Sobald ASTRA Anforderungen hat, die über den Team-Plan hinausgehen, sollte ein Upgrade erwogen werden. Insbesondere:
	•	Identity Management/SSO: Wenn ASTRA alle Nutzerzugänge zentral via z.B. SchweizID/Azure AD steuern will (d.h. bei Austritt automatisch entfernen, Passwortregeln zentral verwalten, etc.), ist SAML SSO und SCIM im Enterprise-Plan ein starkes Argument ￼. Es erhöht Sicherheit und reduziert manuellen Admin-Aufwand. Gerade in einer Bundesbehörde mit möglicherweise wechselnden Projektmitarbeitern wäre dies mittelfristig sinnvoll – aktuell ginge es aber auch, die Zugänge manuell zu pflegen (bei <70 Usern vertretbar).
	•	Erweiterte Compliance: Sollte ASTRA strengere Auditierungsanforderungen bekommen (z.B. Nachvollziehen von Aktivitäten über längere Zeit, Einbindung eines SIEM-Systems via Audit-Log-Streaming) oder die Vorschrift, Code nur in bestimmten Regionen zu hosten, dann bietet Enterprise Cloud die nötigen Features (EU-Datenresidenz, erweiterte Audit-Logs) ￼.
	•	Skalierung und Support: Wenn das ASTRA-Entwicklungsteam deutlich wächst (>>100 Nutzer) oder geschäftskritische Projekte auf GitHub laufen, bei denen ein 99.9% SLA und schneller Hersteller-Support erforderlich sind, ist Enterprise der geeignete Rahmen.

Aktuell jedoch ist ASTRA mit dem Team-Plan gut bedient: Er deckt alle heute benötigten Funktionen ab, insbesondere die für interne Qualitätssicherung so wichtigen Review- und Branchschutz-Features ￼. Die Kosten/Nutzen-Relation ist für die momentane Nutzerzahl deutlich besser als beim Enterprise-Plan. Man erhält Collaboration-Features ohne den Overhead eines Enterprise-Vertrags.

Kosten/Nutzen gegenübergestellt:
	•	GitHub Team: Niedrige laufende Kosten, ausreichende Features für Code- und Dokumentationsmanagement, etwas manueller Benutzerverwaltungsaufwand (aber bei <70 Nutzern machbar), keine SSO. Für ASTRA kurzfristig der wirtschaftlichste Einstieg.
	•	GitHub Enterprise Cloud: Deutlich höhere Kosten (ca. Faktor 5 pro User) ￼ ￼, dafür Komfort und Sicherheit durch SSO, höhere Limits und besseres Audit. Nutzen zieht vor allem, wenn ASTRA die entsprechenden Features aktiv benötigt – was perspektivisch möglich ist (z.B. bei Anschluss ans zentrale IAM des Bundes). Enterprise könnte auch z.B. mit anderen Behörden gemeinsam genutzt werden (Enterprise-Account-Sharing), falls relevant.
	•	GitHub Enterprise Server: Sehr hoher Initial- und Betriebsaufwand, lohnt nur falls Cloud absolut ausgeschlossen. In allen anderen Aspekten kein Mehrwert gegenüber Enterprise Cloud, außer Daten auf eigenem Bunde-Server zu halten. Da GitHub nun EU-Hosting bietet und hohe Sicherheitsstandards, ist Cloud aus unserer Sicht vertretbar.

Fazit: Wir empfehlen ASTRA, zunächst mit GitHub Team zu starten und die Nutzung ein Jahr lang zu evaluieren. Parallel sollte man intern prüfen, ob mittelfristig eine SSO-Integration gewünscht ist. Falls ja, kann frühzeitig mit GitHub ein Wechsel auf Enterprise Cloud vorbereitet werden – GitHub ermöglicht Upgrade der bestehenden Organisation ohne Datenverlust ￼. Dieser schrittweise Ansatz minimiert Kosten, deckt aber die aktuellen Anforderungen vollständig ab.

Durch die Umsetzung der oben beschriebenen Best Practices (strukturierte Repos, PR-Review-Prozess, Rechtekonzept) kann ASTRA so effektiv GitHub nutzen, um sowohl interne Projekte effizient zu entwickeln als auch ausgewählte Ergebnisse transparent mit der Öffentlichkeit zu teilen. Die Entscheidung für das Team- oder Enterprise-Preismodell sollte regelmäßig überprüft werden, während ASTRA’s Digitalisierungsvorhaben wachsen, um sicherzustellen, dass das gewählte Modell weiterhin Kosten/Nutzen-optimal ist und alle benötigten Funktionen bietet.
